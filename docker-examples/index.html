一、为什么需要 Docker

Docker 让软件安装从「装修房子」变成「买家具」——一键部署、删除无痕、迁移方便。

传统方式装一个应用（比如博客），需要安装 Web 服务器、数据库、编程语言环境，然后下载程序、配置环境，出了问题还要查日志、调版本。几年后重装系统，所有步骤重来一遍。

用 Docker 就完全不同了：所有依赖和配置都打包在一个「镜像」里，一条命令就能跑起来，删除也不会在系统里留下垃圾文件。

对比项传统安装Docker
系统影响安装包修改系统，产生垃圾文件容器独立运行，删除即清空
配置复用每台设备重复配置，环境可能不同一份配置文件，任意设备一键部署
应用隔离应用之间相互影响，升级可能冲突容器之间完全隔离，互不干扰
数据管理数据与程序混在一起，备份困难程序在容器，数据在本地，独立管理
系统重装重新安装所有软件和依赖一份配置文件，一条命令恢复
适合人群
• NAS 用户、软路由用户、有闲置电脑的家庭用户
• 想部署媒体中心、笔记、网盘等服务的技术爱好者
• 听过 Docker 但不知道「到底有什么用」的新手

容器 vs 虚拟机

Docker 基于 Linux 内核的 namespace（资源隔离）和 cgroups（资源限制）实现容器化。容器共享宿主机内核，而虚拟机包含完整的 Guest OS。

对比项容器 (Docker)虚拟机 (VM)
启动速度毫秒级分钟级
体积MB 级GB 级
内核共享宿主机内核独立 Guest OS
性能接近原生有虚拟化开销
隔离性进程级隔离系统级隔离
适用场景应用部署、微服务运行不同操作系统
一句话总结：容器是「进程级虚拟化」，共享内核所以轻量；虚拟机是「系统级虚拟化」，独立 OS 所以重量。Docker 的核心价值：一次构建，到处运行。

二、核心概念：6 个必须理解的词

1. 镜像（Image）

应用的「安装包」，但比普通安装包强大得多。

  • 包含应用运行所需的一切：代码、运行环境、依赖库、配置文件
  • 不包含用户数据（数据存在别处）
  • 可以被「拉取」（下载）下来，在任何支持 Docker 的设备上运行
  • 常见来源：Docker Hub（官方镜像仓库）

常见镜像举例：

  • nginx — Nginx 网页服务器
  • wordpress — WordPress 博客程序
  • jellyfin — Jellyfin 媒体服务器
  • portainer/portainer-ce — Docker 可视化管理界面

2. 容器（Container）

镜像「跑起来」之后的实例就叫容器。

  • 一个镜像可以创建无数个容器
  • 容器之间完全隔离，互不影响
  • 容器删除后，容器内的数据也会消失（所以重要数据要存在数据卷里）
  • 容器可以随时启动、停止、重启、删除
镜像 (Image)
→ docker run →
容器 (Container)

3. 仓库（Repository）

存放镜像的地方。可以从仓库拉取（pull）镜像，也可以推送（push）镜像到仓库。

  • Docker Hub — 官方公共仓库，最大的镜像来源
  • 阿里云容器镜像服务 — 国内加速访问
  • GitHub Container Registry — GitHub 的容器仓库
  • 私有仓库 — 企业内部搭建的镜像仓库（如 Harbor）

4. 数据卷（Volume）

容器和本地文件系统之间的「共享文件夹」。

  • 容器删除后，数据卷里的内容不会消失
  • 可以把本地目录挂载到容器里的某个路径
  • 数据库文件、配置文件、用户上传的内容，都应该存在数据卷里
  • 备份数据只需要备份数据卷对应的本地目录

5. 端口映射（Port Mapping）

把容器内部的端口映射到宿主机的端口上，这样外部才能访问容器里的服务。

  • 格式：宿主机端口:容器端口，比如 8080:80 表示把容器的 80 端口映射到宿主机的 8080 端口
  • 宿主机端口不能重复，容器内部端口可以重复
  • 一个容器可以有多个端口映射

6. Docker Compose

用一个 YAML 文件描述一整套服务的配置，一条命令就能启动所有容器。

  • 不用记住一堆 docker run 命令
  • 所有配置写在一个文件里，方便备份和迁移
  • 支持服务之间的网络互通
  • 支持环境变量、数据卷、端口映射等所有配置

三、安装 Docker

Windows 系统

  1. 下载 Docker Desktop 安装包（官网下载）
  2. 双击安装，勾选「WSL 2」选项
  3. 安装完成后重启电脑
  4. 打开 Docker Desktop，等待启动完成
  5. 在设置中开启「Start Docker Desktop when you log in」，开机自启
注意：家庭版 Windows 需要先安装 WSL 2，再安装 Docker Desktop。

Linux 系统（Ubuntu/Debian）

# 一键安装
curl -fsSL https://get.docker.com | sh

# 把当前用户加入 docker 组（不用每次 sudo）
sudo usermod -aG docker $USER

# 注销再登录，然后验证
docker --version
docker run hello-world

macOS

  1. 下载 Docker Desktop for Mac（官网下载）
  2. 拖到 Applications 目录
  3. 启动 Docker Desktop，等待状态栏图标变绿

四、第一个容器：运行 Hello World

# 拉取 hello-world 镜像并运行
docker run hello-world

如果看到 Hello from Docker!，说明安装成功。

这条命令做了什么？

  1. Docker 在本地查找 hello-world 镜像
  2. 没有找到，从 Docker Hub 拉取
  3. 用镜像创建一个容器
  4. 容器运行并输出消息后退出

五、镜像操作

# 搜索镜像
docker search nginx

# 拉取镜像（默认 latest 标签）
docker pull nginx

# 拉取指定版本
docker pull nginx:1.25

# 查看本地镜像
docker images

# 删除镜像
docker rmi nginx:1.25

# 清理无用镜像
docker image prune -f
最佳实践：指定具体的版本号（如 nginx:1.25），不要用 latest。latest 不可控，可能有 bug 或不兼容。

六、容器操作

基本操作

# 后台运行容器
docker run -d --name my-nginx -p 8080:80 nginx

# 参数说明：
# -d              后台运行（detach）
# --name my-nginx 给容器起个名字
# -p 8080:80      端口映射（宿主机:容器）

# 查看运行中的容器
docker ps

# 查看所有容器（包括停止的）
docker ps -a

# 停止容器
docker stop my-nginx

# 启动已停止的容器
docker start my-nginx

# 重启容器
docker restart my-nginx

# 删除容器
docker rm my-nginx

进入容器

# 进入运行中的容器（bash）
docker exec -it my-nginx bash

# 退出容器
exit

# 查看容器日志
docker logs my-nginx

# 实时查看日志
docker logs -f my-nginx

数据挂载示例

# 挂载本地目录到容器
docker run -d \
  --name my-nginx \
  -p 8080:80 \
  -v ./html:/usr/share/nginx/html:ro \
  nginx:1.25

容器生命周期

Created
→ docker start →
Running
→ docker stop →
Stopped
→ docker rm →
Deleted
# 创建容器（不启动）
docker create --name my-nginx nginx:1.25

# 启动已创建的容器
docker start my-nginx

# docker run = create + start + attach
docker run -d --name my-nginx nginx:1.25

# 停止容器（发送 SIGTERM，超时后 SIGKILL）
docker stop my-nginx

# 强制停止（直接 SIGKILL）
docker kill my-nginx

# 删除已停止的容器
docker rm my-nginx

# 强制删除运行中的容器
docker rm -f my-nginx

宿主机与容器间复制文件

# 宿主机 → 容器
docker cp ./config.json my-nginx:/app/config.json

# 容器 → 宿主机
docker cp my-nginx:/var/log/nginx/access.log ./logs/

资源限制（cgroups）

通过 docker run 参数限制容器可使用的资源，防止某个容器吃满宿主机。

# 限制内存
docker run -d --name my-app \
  --memory=512m \
  --memory-swap=512m \
  myapp:1.0

# 限制 CPU
docker run -d --name my-app \
  --cpus=1.5 \
  --cpu-shares=512 \
  myapp:1.0

# 限制磁盘 IO
docker run -d --name my-app \
  --device-read-bps /dev/sda:10mb \
  --device-write-bps /dev/sda:10mb \
  myapp:1.0

# 重启策略
docker run -d --name my-app \
  --restart unless-stopped \
  myapp:1.0
参数说明
--memory=512m最大内存 512MB
--memory-swap=512m内存 + swap 总量（设为 -1 不限 swap）
--cpus=1.5最多使用 1.5 个 CPU 核心
--cpu-shares=512CPU 份额（相对权重，默认 1024）
--restart no不自动重启（默认）
--restart always总是重启
--restart unless-stopped除非手动停止，否则自动重启（推荐）
--restart on-failure:3失败时重启，最多 3 次

七、数据卷

数据卷是容器删除后仍能保留数据的机制。

# 创建数据卷
docker volume create my-data

# 查看数据卷
docker volume ls

# 使用数据卷
docker run -d \
  --name my-postgres \
  -e POSTGRES_PASSWORD=secret \
  -v my-data:/var/lib/postgresql/data \
  postgres:16

# 查看数据卷详情
docker volume inspect my-data

# 删除数据卷
docker volume rm my-data

三种挂载方式对比

类型说明数据持久性适用场景
具名卷由 Docker 管理，数据存储在 Docker 目录容器删除后保留数据库数据
绑定挂载直接映射宿主机目录或文件容器删除后保留配置文件、代码
tmpfs存储在内存中，容器重启后丢失容器重启后丢失临时文件、敏感数据
# 具名卷
docker run -v my-data:/var/lib/postgresql/data postgres:16

# 绑定挂载（绝对路径）
docker run -v /host/path:/container/path nginx

# 绑定挂载（相对路径，相对于当前目录）
docker run -v ./config:/etc/nginx/nginx.conf:ro nginx

# tmpfs（内存中存储，适合临时文件）
docker run --tmpfs /tmp:rw,noexec,nosuid,size=100m nginx

八、网络

Docker 提供四种网络模式，容器之间通过网络通信。

四种网络模式

模式说明适用场景
bridge（默认）容器通过 Docker 网桥（docker0）与宿主机通信，容器间可互通大多数场景
host容器直接使用宿主机网络栈（无隔离，性能最高）高性能网络场景
none无网络，完全隔离安全敏感场景
container:<name>复用另一个容器的网络栈共享网络的 sidecar
# bridge 模式（默认）
docker run -d --name web nginx

# host 模式（直接用宿主机网络）
docker run -d --name web --network host nginx

# none 模式（无网络）
docker run -d --name web --network none nginx

# container 模式（共享另一个容器的网络）
docker run -d --name sidecar --network container:web myapp

自定义网络

# 创建自定义网络
docker network create my-net

# 查看网络
docker network ls

# 运行容器并加入网络
docker run -d --name redis --network my-net redis:alpine
docker run -d --name my-app --network my-net myapp:1.0

# 同一网络内的容器可以用容器名互相访问
# my-app 中连接 redis:6379 即可

# 将容器连接到多个网络
docker network connect my-net my-app

# 断开网络
docker network disconnect my-net my-app
容器间通信：同一 Docker 网络中的容器可以通过容器名或服务名互相访问，不需要用 IP 地址。自定义网络支持 DNS 自动解析，方便微服务通信。

九、Dockerfile 详解

Dockerfile 是一个文本文件，由一系列指令组成，描述了如何从基础镜像一步步构建出自定义镜像。Docker 会按顺序读取每条指令，每条指令创建一个「层」（layer），最终叠加成完整的镜像。

9.1 什么是镜像分层

Docker 镜像是由多个只读层叠加而成的。每一层都是前一层基础上的增量变化。例如：

基础镜像层
node:20-alpine
+
npm install 层
依赖安装
+
COPY 代码层
应用代码
=
最终镜像
my-app:1.0

分层的好处：

  • 缓存复用：如果某一层没变化，构建时直接复用缓存，大幅加速构建
  • 节省空间：多个镜像共享相同的基础层，磁盘占用更少
  • 传输高效：拉取/推送时只传输变化的层

9.2 完整指令详解

FROM — 指定基础镜像

每个 Dockerfile 必须以 FROM 开头，指定一个基础镜像。后续所有操作都在这个基础上进行。

# 官方镜像，带版本号
FROM node:20-alpine

# 使用更小的基础镜像（Debian slim 变体）
FROM python:3.11-slim

# 多阶段构建：第一阶段编译，第二阶段只拷贝产物
FROM golang:1.22 AS builder
WORKDIR /src
COPY . .
RUN go build -o /app .

FROM alpine:3.19
COPY --from=builder /app /app
CMD ["/app"]
镜像命名规则：镜像名:标签。不写标签默认 latest。Alpine 变体体积最小（~5MB），Debian slim 次之（~80MB），完整 Debian 最大（~120MB）。

WORKDIR — 设置工作目录

后续所有 COPY、RUN、CMD 指令都会在这个目录下执行。如果目录不存在会自动创建。

# 设置工作目录
WORKDIR /app

# 可以多次使用，切换不同目录
WORKDIR /src
COPY . .
WORKDIR /app
RUN npm install

COPY — 复制文件

将宿主机的文件复制到镜像中。

# 复制单个文件
COPY package.json ./

# 复制目录（src/ 下的所有内容复制到 /app/src/）
COPY src/ ./src/

# 复制多个文件
COPY *.js ./

# 使用通配符
COPY package*.json ./

# 复制并设置所有者（UID:GID）
COPY --chown=1000:1000 . .

# 从另一个镜像复制（多阶段构建）
COPY --from=builder /app/dist ./dist
COPY vs ADD：COPY 只做简单复制；ADD 额外支持自动解压 tar 包和远程 URL。推荐用 COPY，行为更明确。

ADD — 复制文件（增强版）

# 自动解压 tar 包
ADD app.tar.gz /usr/src/

# 从远程 URL 下载（不推荐，建议用 RUN curl）
ADD https://example.com/file.zip /tmp/

RUN — 执行命令

在构建过程中执行任意命令，结果会固化到镜像层中。常用于安装依赖、编译代码等。

# shell 格式（通过 /bin/sh -c 执行）
RUN apt-get update && apt-get install -y \
    curl \
    git \
    && rm -rf /var/lib/apt/lists/*

# exec 格式（直接执行，不经过 shell）
RUN ["npm", "install", "--production"]

# 安装 Python 依赖
RUN pip install --no-cache-dir -r requirements.txt

# 多条 RUN 合并为一条（减少镜像层数）
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        build-essential \
        libpq-dev \
    && rm -rf /var/lib/apt/lists/* \
    && pip install --no-cache-dir -r requirements.txt
shell 格式 vs exec 格式
• shell 格式：RUN apt-get update — 等价于 /bin/sh -c "apt-get update"，支持变量替换、管道等
• exec 格式：RUN ["apt-get", "update"] — 直接执行，不经过 shell，适合不需要 shell 特性的场景

CMD — 容器启动命令

指定容器启动时执行的命令。一个 Dockerfile 只能有一条 CMD（最后一条生效）。会被 docker run 的参数覆盖。

# exec 格式（推荐）
CMD ["node", "server.js"]

# shell 格式
CMD node server.js

# 作为 ENTRYPOINT 的默认参数
CMD ["--port", "8080"]

ENTRYPOINT — 入口点

与 CMD 类似，但不会被 docker run 的参数覆盖，而是追加。适合需要固定执行某个程序的场景。

# 固定执行 python，docker run myimage script.py 会变成 python script.py
ENTRYPOINT ["python"]

# 配合 CMD 提供默认参数
ENTRYPOINT ["python"]
CMD ["app.py"]
CMD vs ENTRYPOINT 对比
• CMD：容器启动的默认命令，可被 docker run 参数覆盖
• ENTRYPOINT：容器启动的固定入口，docker run 参数会追加到后面
• 组合使用：ENTRYPOINT 指定程序，CMD 指定默认参数

ENV — 环境变量

# 设置环境变量
ENV NODE_ENV=production
ENV APP_PORT=3000

# 可用 ${变量名} 引用
ENV APP_HOME=/app
WORKDIR ${APP_HOME}

# 批量设置
ENV POSTGRES_HOST=localhost \
    POSTGRES_PORT=5432 \
    POSTGRES_DB=mydb

ARG — 构建参数

只在构建过程中有效，容器运行时不存在。可用于传递版本号等构建时参数。

# 定义构建参数
ARG NODE_VERSION=20

FROM node:${NODE_VERSION}-alpine

# 构建时传入：docker build --build-arg NODE_VERSION=18 .

EXPOSE — 声明端口

声明容器运行时监听的端口（仅文档作用，不会自动发布端口）。实际发布仍需 -p 参数。

EXPOSE 80
EXPOSE 443
EXPOSE 3000/udp

USER — 指定运行用户

指定后续 RUN、CMD、ENTRYPOINT 以哪个用户身份运行。安全最佳实践：不用 root。

# 创建非 root 用户
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# 切换用户
USER appuser

CMD ["node", "server.js"]

HEALTHCHECK — 健康检查

告诉 Docker 如何判断容器是否健康。Docker 会定期执行该命令，根据返回码判断状态。

# 每 30 秒检查一次，超时 10 秒，连续 3 次失败标记为 unhealthy
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# 禁用基础镜像的健康检查
HEALTHCHECK NONE

VOLUME — 匿名卷

声明容器中应作为数据卷的目录。即使不指定 -v，运行时也会自动创建匿名卷。

VOLUME /var/lib/postgresql/data
VOLUME /data

LABEL — 元数据标签

LABEL maintainer="dev@example.com"
LABEL version="1.0"
LABEL description="My awesome app"

.dockerignore — 排除文件

类似于 .gitignore，排除不需要发送到构建上下文的文件，减小上下文体积。

# .dockerignore
node_modules
.git
.env
*.md
dist
coverage
.DS_Store

9.3 实战示例

示例 1：Node.js 应用（生产级）

# ---- 构建阶段 ----
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# ---- 运行阶段 ----
FROM node:20-alpine
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=builder /app/dist ./dist
USER app
EXPOSE 3000
CMD ["node", "dist/server.js"]

示例 2：Python Flask 应用

FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .
EXPOSE 5000
CMD ["gunicorn", "-w", "4", "-b", "0.0.0.0:5000", "app:app"]

示例 3：Nginx 静态站点

FROM nginx:1.25-alpine
COPY nginx.conf /etc/nginx/nginx.conf
COPY dist/ /usr/share/nginx/html/
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

示例 4：Go 应用（多阶段构建）

9.5 最佳实践总结

Dockerfile 最佳实践

1. 利用构建缓存：先 COPY 依赖文件（package.json），再 RUN 安装，最后 COPY 源代码。依赖不变则缓存命中。

2. 使用多阶段构建：编译阶段和运行阶段分离，最终镜像只包含运行所需文件，体积更小。

3. 选择轻量基础镜像：优先 Alpine（~5MB），次选 slim（~80MB），避免使用 ubuntu/debian 完整版。

4. 合并 RUN 指令：多个 RUN 合并为一条，减少镜像层数。末尾清理缓存（rm -rf /var/lib/apt/lists/*）。

5. 不要用 root 运行：创建专用用户 RUN addgroup -S app && adduser -S app -G app，最后 USER app。

6. 使用 .dockerignore：排除 node_modules、.git、.env 等，减小构建上下文体积。

7. 固定版本号：FROM node:20.11-alpine 比 FROM node:latest 更可控。

8. 一个进程一个容器：每个容器只运行一个主进程，日志输出到 stdout/stderr。

十、Docker Compose 详解

Docker Compose 是一个用于定义和运行多容器应用的工具。你只需一个 YAML 文件（compose.yaml 或 docker-compose.yml），就能描述所有服务、网络、卷的配置，然后用一条命令启动、停止、管理整个应用栈。

10.1 命令式 vs 声明式

方式说明示例
命令式告诉系统「先做A，再做B，如果出错则C」shell 脚本、docker run 命令
声明式告诉系统「我需要什么状态」，Docker 自己决定怎么实现compose.yaml

声明式的好处：可读性好、可复现、可维护、版本控制友好。

10.4 services（服务）详解

每个 service 代表一个容器角色。以下是所有常用字段：

services:
  web:
    # ---- 镜像来源 ----
    image: nginx:1.25                    # 直接使用镜像
    build: ./app                         # 用 Dockerfile 构建
    build:
      context: ./app                     # 构建上下文目录
      dockerfile: Dockerfile             # 指定 Dockerfile（默认 Dockerfile）
      args:                              # 构建参数
        NODE_VERSION: 20
      cache_from:                        # 构建缓存来源
        - my-app:latest

    # ---- 端口映射 ----
    ports:
      - "8080:80"                        # 宿主机:容器
      - "443:443"
      - "127.0.0.1:3000:3000"           # 只绑定 localhost
      - "9090:9090/udp"                 # UDP 端口

    # ---- 数据卷 ----
    volumes:
      - ./html:/usr/share/nginx/html:ro  # 绑定挂载（只读）
      - nginx-cache:/var/cache/nginx     # 具名卷

    # ---- 环境变量 ----
    environment:
      NODE_ENV: production
      DATABASE_URL: postgres://db:5432/mydb
    env_file:
      - .env                             # 从文件加载
      - .env.local                       # 可以多个文件

    # ---- 依赖管理 ----
    depends_on:
      - redis
      - db

    # ---- 重启策略 ----
    restart: unless-stopped
    # no | always | on-failure | unless-stopped

    # ---- 健康检查 ----
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s                      # 检查间隔
      timeout: 10s                       # 超时时间
      retries: 3                         # 重试次数
      start_period: 40s                  # 启动宽限期
      start_interval: 5s                 # 启动期间检查频率（2026新特性）

    # ---- 资源限制 ----
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: "1.0"
        reservations:
          memory: 256M
          cpus: "0.5"

    # ---- 网络 ----
    networks:
      - frontend
      - backend

    # ---- 命令覆盖 ----
    command: ["node", "server.js"]       # 覆盖 CMD
    entrypoint: ["/entrypoint.sh"]       # 覆盖 ENTRYPOINT
    working_dir: /app                    # 覆盖 WORKDIR
    user: "1000:1000"                    # 覆盖 USER
    hostname: web-server
    domainname: example.com
    stdin_open: true                     # 等价 docker run -i
    tty: true                            # 等价 docker run -t
    read_only: true                      # 只读文件系统
    tmpfs:                               # 临时文件系统
      - /tmp
      - /var/run
    logging:                             # 日志配置
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

10.5 volumes（卷）详解

services:
  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data   # 具名卷
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql:ro  # 绑定挂载
      - pgdata:/var/log/postgresql        # 同一卷挂载到多个位置

  nginx:
    image: nginx:1.25
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro    # 只读配置
      - ./ssl:/etc/nginx/ssl:ro                  # SSL 证书
      - static-files:/usr/share/nginx/html       # 静态文件

volumes:
  pgdata:                                  # 顶层声明
    driver: local                          # 驱动（默认 local）
  static-files:

三种卷类型对比

类型语法特点适用场景
具名卷name:/pathDocker 管理，数据在 Docker 目录数据库、持久数据
匿名卷:/pathDocker 自动生成随机名临时数据（不推荐）
绑定挂载./host:/path直接映射宿主机目录配置文件、代码

卷的读写权限

volumes:
  # 默认：读写
  - ./data:/app/data

  # 只读（容器不能写入）
  - ./config:/app/config:ro

  # 指定用户 ID（解决权限问题）
  - ./data:/app/data:rw,U=1000:1000

10.6 networks（网络）详解

services:
  proxy:
    image: nginx
    networks:
      - frontend

  app:
    image: myapp
    networks:
      - frontend
      - backend

  db:
    image: postgres:16
    networks:
      - backend

  cache:
    image: redis:alpine
    networks:
      backend:
        aliases:
          - redis        # 可以用 redis 或 cache 访问

networks:
  frontend:              # 前端网络（proxy 和 app 可通信）
    driver: bridge       # 默认驱动
  backend:               # 后端网络（app、db、cache 可通信）
    driver: bridge

网络隔离效果：

proxy
frontend
↔
app
frontend + backend
↔
db
backend
↔
cache
backend

proxy 只能访问 app；app 可以访问 proxy 和 db/cache；db 只能被 app 和 cache 访问。

10.7 depends_on 详解

services:
  web:
    image: myapp
    depends_on:
      # 短语法：只等启动
      - redis

      # 长语法：等健康检查通过
      db:
        condition: service_healthy

      # 等一次性任务完成
      migrations:
        condition: service_completed_successfully

  redis:
    image: redis:alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 3

  db:
    image: postgres:16
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "postgres"]
      interval: 5s
      timeout: 3s
      retries: 5

  migrations:
    image: myapp:latest
    command: ["python", "manage.py", "migrate"]
    restart: "no"
depends_on 的三个 condition
• service_started（默认）：容器启动即可，不等就绪
• service_healthy：等健康检查通过
• service_completed_successfully：等容器正常退出（适合一次性任务）

10.8 环境变量管理

# ---- .env 文件（Compose 变量替换） ----
POSTGRES_VERSION=16
APP_PORT=8080
REDIS_PASSWORD=secret123
DOMAIN=example.com

# ---- compose.yaml ----
services:
  db:
    image: postgres:${POSTGRES_VERSION}     # 用 .env 变量替换镜像版本
    environment:
      POSTGRES_PASSWORD: ${REDIS_PASSWORD}  # 注入容器环境变量
      POSTGRES_DB: mydb

  web:
    image: nginx:1.25
    ports:
      - "${APP_PORT}:80"                    # 端口也用变量
    env_file:
      - .env                                # 把 .env 中所有变量注入容器
      - .env.production                     # 可以加载多个文件

env_file 格式

# .env 文件（简单格式）
POSTGRES_VERSION=16
APP_PORT=8080

# .env.local 文件（带引号）
DATABASE_URL="postgres://user:pass@db:5432/mydb"
SECRET_KEY="my-secret-key"
变量优先级：环境变量 > .env 文件 > Compose 文件中的默认值。shell 中 export APP_PORT=9090 会覆盖 .env 中的值。

10.13 Dockerfile 与 Compose 的关系

Dockerfile 解决「镜像怎么构建」；Compose 解决「容器怎么组织和运行」。

十三、常见问题与踩坑

坑 1：数据没有挂载，容器删除后数据丢失
这是新手最容易犯的错误。容器删除后，容器内部的所有数据都会消失。

解法：数据库文件、配置文件、用户上传的数据，一定要用 volumes 挂载到本地目录。备份的时候只需要备份本地目录。
坑 2：端口冲突，容器启动失败
宿主机的某个端口已经被其他程序占用了，容器就启动不起来。

解法：看容器日志，报 port is already allocated 就是端口冲突。换一个宿主机端口就行。
坑 3：镜像拉取失败，网络问题
国内访问 Docker Hub 有时候会很慢，甚至拉取失败。

解法：配置国内镜像加速（阿里云、腾讯云都有免费的镜像加速服务）。
坑 4：depends_on 不等于「就绪」
depends_on 只保证启动顺序，不保证服务真的就绪。比如 postgres 容器启动了但还没完成初始化，应用连上去就报连接拒绝。

解法：加健康检查 + 条件等待：condition: service_healthy。
坑 5：容器间用 localhost 互相访问
每个容器有自己的 localhost。server 容器里写 localhost:5432 连 postgres 是连不上的。

解法：同 Compose 项目内的容器，用服务名互访：postgres:5432。Compose 会自动创建 DNS 解析。
坑 6：ports 用数字不加引号
YAML 里 ports: - 80:80，YAML 会把 80:80 解析成时间字符串。

解法：端口映射永远用字符串 "80:80"。
坑 7：docker compose down --volumes 删数据
docker compose down 默认只删容器和网络，不动 volume。但加了 --volumes 参数，数据卷也一起删了。

解法：永远不加 --volumes，除非你明确知道自己在做什么。
一句话原则：只要超过一个容器，就用 Compose。哪怕只有一个容器，Compose 也比 docker run 好维护。
