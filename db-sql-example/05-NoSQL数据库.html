05 - NoSQL 数据库

NoSQL — Not Only SQL

NoSQL 数据库是一类非关系型数据库的总称，专为大数据量、高并发、灵活数据模型而设计。

为什么需要 NoSQL？

关系数据库的局限性催生了 NoSQL

  • 扩展困难 — RDBMS 纵向扩展（升级硬件）有上限，横向扩展复杂
  • 表结构固定 — Schema 变更需要迁移，灵活性不足
  • 大数据性能 — 复杂 JOIN 在海量数据下性能下降
  • 非结构化数据 — JSON、日志、社交关系不适合表格存储

NoSQL 的五大类型

不同数据模型适用于不同场景

类型 数据模型 代表产品 典型场景
文档型 JSON / BSON 文档 MongoDB, Couchbase CMS、博客、日志、电商
键值型 Key-Value 对 Redis, DynamoDB 缓存、会话、购物车
列族型 行+列族 Cassandra, HBase 时序、IoT、日志分析
图数据库 节点+边 Neo4j, JanusGraph 社交、推荐、风控

文档数据库 — MongoDB

以 JSON 文档形式存储数据，灵活可嵌套

// MongoDB 文档示例
{
  "_id": "user_001",
  "name": "张三",
  "email": "zhangsan@example.com",
  "address": {
    "city": "北京",
    "street": "朝阳路 100 号"
  },
  "orders": [
    { "order_id": "O001", "amount": 1500 },
    { "order_id": "O002", "amount": 800 }
  ]
}

// 查询操作
db.users.find({ "address.city": "北京" })
db.users.find({ "orders.amount": { $gt: 1000 } })
文档数据库适合数据结构多变、有嵌套关系的场景。MongoDB 是当前最流行的文档数据库。

键值数据库 — Redis

基于内存的键值存储，极高性能

# Redis 命令示例
SET user:1001 "张三"
GET user:1001
SET session:abc123 '{"user_id":1001,"expire":3600}'

# 支持多种数据结构
LPUSH queue task1         # 列表
SADD tags:react "frontend"  # 集合
ZADD leaderboard 100 "Alice"  # 有序集合
EXPIRE session:abc123 3600   # 设置过期时间
Redis 速度快（微秒级），常用于缓存、分布式锁、计数器、排行榜等场景。也支持持久化。

列族数据库 — Cassandra

面向列的宽表存储，适合大规模写入

-- Cassandra CQL
CREATE TABLE user_events (
    user_id   UUID,
    event_time TIMESTAMP,
    event_type TEXT,
    payload   TEXT,
    PRIMARY KEY (user_id, event_time)
);

-- 按用户查询事件
SELECT * FROM user_events
WHERE user_id = 123e4567-e89b-12d3-a456-426614174000';
Cassandra 没有单点故障，支持跨数据中心复制，适合物联网、日志、时序数据。

图数据库 — Neo4j

用节点和关系表示高度关联的数据

-- Neo4j Cypher 查询
// 创建节点和关系
CREATE (a:Person {name: 'Alice'})
CREATE (b:Person {name: 'Bob'})
CREATE (a)-[:FRIENDS_WITH]->(b)

// 查询好友的好友
MATCH (a:Person {name: 'Alice'})
      -[:FRIENDS_WITH]->()
      -[:FRIENDS_WITH]->(fof)
RETURN fof.name
图数据库将关系作为一等公民，遍历关系的性能远优于关系数据库的多层 JOIN。

RDBMS vs NoSQL 对比

维度 RDBMS NoSQL
数据模型 表、行、列，严格 Schema 文档/键值/列族/图，灵活 Schema
扩展方式 纵向扩展为主（升级硬件） 横向扩展为主（增加节点）
ACID 事务 完整支持 通常支持 BASE（最终一致性）
查询语言 SQL 标准 各产品自己的查询语言
适合场景 传统业务、金融、ERP 大数据、实时、IoT、社交
数据一致性 强一致 最终一致（部分支持强一致）

如何选择？

选择数据库的原则

  • 数据结构固定、需要复杂关联 → PostgreSQL / MySQL
  • 数据结构多变、嵌套深 → MongoDB
  • 极速读写、缓存场景 → Redis
  • 海量写入、时序数据 → Cassandra / InfluxDB
  • 高度关联、关系遍历 → Neo4j
  • 全文搜索和分析 → Elasticsearch
现代架构常采用"混合持久化"（Polyglot Persistence），不同场景使用最合适的数据库。

小结

  • NoSQL = Not Only SQL，是为解决 RDBMS 局限而生
  • 文档 / 键值 / 列族 / 图 四种主要类型
  • NoSQL 通常支持横向扩展和灵活 Schema
  • NoSQL 牺牲强一致性换取性能和扩展性（BASE）
  • 实际项目常采用多种数据库组合使用
