08 - 数据库最佳实践

让数据库高效、安全、可靠

数据库是大多数应用的基石，良好的实践能显著提升性能、安全性和可维护性。

SQL 编写规范

好的 SQL 风格让团队协作更高效

  • 关键字大写 — SELECT, FROM, WHERE, JOIN 大写，增强可读性
  • 表名/列名小写 — 使用下划线命名：user_account
  • 明确列名 — 避免 SELECT *，只取需要的列
  • 使用别名 — 多表查询时用简短别名 u, o
  • 格式化 — 每行一个子句，逗号前置方便注释
-- 推荐风格
SELECT u.name,
       o.order_id,
       o.amount,
       o.order_date
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.amount >= 100
  AND u.status = 'active'
ORDER BY o.amount DESC
LIMIT 20;

索引策略

索引是性能优化的核心

  • 为 WHERE 条件中的列建索引 — 最频繁的查询条件优先
  • 复合索引 — 多列条件时，最左前缀原则
  • 不要过度索引 — 索引占空间，写操作会变慢
  • 监控慢查询 — 用 EXPLAIN ANALYZE 分析执行计划
-- 查看慢查询（PostgreSQL）
SELECT query, calls, total_exec_time, rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 5;

-- 分析查询
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 123
  AND status = 'completed';

查询优化技巧

常见优化手段

  • 尽量使用覆盖索引 — 索引包含所有查询的列，避免回表
  • 分页优化 — 用 WHERE id > last_id LIMIT 20 替代 OFFSET
  • 避免 SELECT * — 只取需要的列，减少 I/O
  • 合理使用 JOIN — 多表 JOIN 时确保关联列有索引
  • 子查询 vs JOIN — 通常 JOIN 性能更好，但视具体情况而定
-- ❌ 低效分页
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 100000;

-- ✅ 高效分页（游标分页）
SELECT * FROM orders WHERE id > 100000 ORDER BY id LIMIT 20;

-- ❌ SELECT *
SELECT * FROM users WHERE email = 'test@example.com';

-- ✅ 覆盖索引查询
SELECT id, name, email FROM users WHERE email = 'test@example.com';

安全最佳实践

数据库安全是不可忽视的一环

  • 最小权限原则 — 每个用户只授予需要的最小权限
  • 密码加密 — 不存储明文密码，使用 bcrypt/argon2 哈希
  • 参数化查询 — 永远使用占位符，防止 SQL 注入
  • 网络隔离 — 数据库不暴露在公网，仅内网访问
  • SSL/TLS — 数据库连接使用加密传输
-- 权限管理
CREATE USER readonly_user WITH PASSWORD 'strong_pass';
GRANT CONNECT ON DATABASE myapp TO readonly_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_user;
REVOKE INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public FROM readonly_user;

备份与容灾

确保数据不丢失

策略 说明 频率
全量备份 导出整个数据库 每天
增量备份 只备份变更的数据（WAL 归档） 持续/每小时
主从复制 实时同步到从库，故障时切换 实时
异地容灾 跨地域存储备份 每天

运维监控

日常运维检查清单

-- 检查数据库大小
SELECT pg_database_size('mydb') / 1024 / 1024 AS size_mb;

-- 检查表大小
SELECT
    relname AS table_name,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 10;

-- 查看当前连接
SELECT state, COUNT(*) FROM pg_stat_activity
WHERE datname = 'mydb'
GROUP BY state;

-- 查看锁等待
SELECT blocked_locks.pid     AS blocked_pid,
       blocked_activity.usename  AS blocked_user,
       blocking_locks.pid     AS blocking_pid,
       blocking_activity.usename AS blocking_user
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
WHERE NOT blocked_locks.granted;

数据库选型指南

如何为项目选择合适的数据库

业务场景 推荐数据库 理由
电商/金融/ERP PostgreSQL ACID 事务、数据完整性
博客/CMS/社交 MySQL / MongoDB 生态成熟、文档丰富
缓存/会话/实时 Redis 微秒级延迟、丰富的数据结构
搜索/日志分析 Elasticsearch 全文搜索、聚合分析
IoT/时序数据 TimescaleDB / InfluxDB 时序优化、压缩存储
社交图谱/推荐 Neo4j 关系遍历高效

学习路径建议

继续深入的方向

  • 深入数据库原理：B+ 树、MVCC、查询优化器
  • 分布式数据库：TiDB、CockroachDB、Spanner
  • 数据仓库与 OLAP：ClickHouse、StarRocks、Snowflake
  • 流处理与实时：Kafka + Flink、Debezium 变更数据捕获
  • 数据库运维：pgAdmin、Prometheus + Grafana 监控

小结

  • 规范的 SQL 风格和索引策略是性能优化的基础
  • 参数化查询和最小权限是安全底线
  • 备份、容灾、监控是运维的三大支柱
  • 根据业务场景选择合适的数据库
  • 祝贺你完成了数据库与 SQL 的入门学习！
