04 - 关系数据库设计

好的设计让数据更可靠

关系数据库设计的核心是数据建模、范式化和保持数据完整性。良好的设计可以减少冗余、避免异常。

关系型数据模型

关系数据库以二维表的形式组织数据

概念 说明
关系（Relation） 即一张二维表
元组（Tuple） 表中的一行
属性（Attribute） 表中的一列
域（Domain） 属性的取值范围
键（Key） 唯一标识元组的属性集 
-- 关系模型示例：学生表
CREATE TABLE Student (
    student_id   CHAR(8)  PRIMARY KEY,         -- 学号（主键）
    name         VARCHAR(50) NOT NULL,          -- 姓名
    gender       CHAR(1) CHECK (gender IN ('M','F')),  -- 性别（域约束）
    birth_date   DATE,
    department   VARCHAR(50)
);

-- 课程表
CREATE TABLE Course (
    course_id    CHAR(6) PRIMARY KEY,
    title        VARCHAR(100) NOT NULL,
    credits      INTEGER CHECK (credits > 0)
);

-- 选课表（多对多关系）
CREATE TABLE Enrollment (
    student_id   CHAR(8) REFERENCES Student(student_id),
    course_id    CHAR(6) REFERENCES Course(course_id),
    grade        DECIMAL(3,1),
    PRIMARY KEY (student_id, course_id)          -- 复合主键
);

实体关系模型 (ER)

使用 ER 图进行概念设计

  • 实体（Entity） — 现实世界中的对象（学生、课程）
  • 属性（Attribute） — 实体的特征（姓名、学分）
  • 关系（Relationship） — 实体之间的关联（选课）
  • 关系类型：
    • 1:1（一对一）— 一个学生一个学号
    • 1:N（一对多）— 一个部门多个学生
    • M:N（多对多）— 一个学生选多门课，一门课被多个学生选

数据库范式 (Normalization)

范式是评估表设计的标准，目的是减少数据冗余和更新异常

第一范式 (1NF)

每个列都是不可分割的原子值

-- ❌ 违反 1NF：phone 包含多个电话号码
CREATE TABLE Customer (
    customer_id INT PRIMARY KEY,
    phones       VARCHAR(200)  -- "13800138000,13900139000"
);

-- ✅ 符合 1NF：每个电话一行
CREATE TABLE CustomerPhone (
    customer_id INT REFERENCES Customer(customer_id),
    phone       VARCHAR(20),
    PRIMARY KEY (customer_id, phone)
);

第二范式 (2NF)

在 1NF 基础上，消除非主属性对候选键的部分函数依赖

-- ❌ 违反 2NF：订单明细表，product_name 只依赖于 product_id
CREATE TABLE OrderDetail (
    order_id    INT,
    product_id  INT,
    product_name VARCHAR(100),  -- 部分依赖
    quantity    INT,
    PRIMARY KEY (order_id, product_id)
);

-- ✅ 符合 2NF：拆成两张表
CREATE TABLE OrderDetail (
    order_id   INT,
    product_id INT,
    quantity   INT,
    PRIMARY KEY (order_id, product_id)
);
-- Product 表单独存放 product_name

第三范式 (3NF)

在 2NF 基础上，消除传递依赖

-- ❌ 违反 3NF：department_name 依赖于 department_id
CREATE TABLE Employee (
    emp_id          INT PRIMARY KEY,
    name            VARCHAR(100),
    department_id   INT,
    department_name VARCHAR(100)  -- 传递依赖
);

-- ✅ 符合 3NF：部门信息独立成表
CREATE TABLE Employee (
    emp_id        INT PRIMARY KEY,
    name          VARCHAR(100),
    department_id INT REFERENCES Department(department_id)
);

CREATE TABLE Department (
    department_id   INT PRIMARY KEY,
    department_name VARCHAR(100)
);
通常达到 3NF 已经满足大部分业务需求。BCNF、4NF、5NF 在实际中较少用到。

ACID 特性

关系数据库事务的四大保证

特性 说明 示例
A 原子性 事务要么全部成功，要么全部失败 转账时扣款和入账必须同时完成
C 一致性 事务前后数据保持一致约束 转账后双方余额总和不变
I 隔离性 并发事务互不干扰 两个用户同时转账结果正确
D 持久性 提交的事务永久保存 断电重启后数据不丢失

事务控制

用 TCL 语句管理事务

-- 开始事务
BEGIN;

-- 转账操作：A 扣钱
UPDATE Account SET balance = balance - 1000
WHERE account_id = 'A001';

-- B 加钱
UPDATE Account SET balance = balance + 1000
WHERE account_id = 'B001';

-- 确认提交
COMMIT;

-- 如果中途出错，可以回滚
ROLLBACK;

-- 设置保存点
SAVEPOINT sp1;
-- ... 某些操作
ROLLBACK TO sp1;

小结

  • ER 模型是概念设计工具，关系模型是逻辑设计基础
  • 1NF → 2NF → 3NF 逐步消除数据冗余和更新异常
  • ACID 是关系数据库事务的四大特性保证
  • 事务用 BEGIN/COMMIT/ROLLBACK 控制
  • 良好的数据库设计是应用稳定运行的基石
